# 《文字狱》— 游戏原型设计文档

## 一、游戏概述

**类型**：回合制文字解谜 + 迷宫探索  
**引擎**：bracket-lib (Rust)  
**核心概念**：你控制一个角色在地图上移动，但**不能直接操控角色移动方向**——你只能通过键盘打字来下达指令。**每个字符消耗 1 步**，方向指令本身也由字符拼出，同样消耗步数。你所打的每一个字符都会**永远留在地面上**，成为**不可逾越的永久障碍**——地图只會越走越小。

## 二、核心机制

### 2.1 角色状态

| 属性 | 说明 |
|------|------|
| **位置** | (x, y) 网格坐标 |
| **朝向** | 右 (默认) / 左 / 上 / 下 |
| **剩余步数** | 初始由关卡指定，归零则失败 |
| **历史栈** | 记录每一步的位置和输入，供 backspace 回溯 |

### 2.2 角色显示

角色始终显示为当前朝向的**方向箭头**：

```
→   朝右
↑   朝上
←   朝左
↓   朝下
```

### 2.3 输入规则

```
每一帧轮询键盘输入，即时执行：

1. 方向指令
   输入字符累积拼出方向词，完整匹配后改变朝向。
   方向词字符留在当前格子（不移动）。
   方向词（及消耗）：
     up    → 朝上 (2步)
     down  → 朝下 (4步)
     left  → 朝左 (4步)
     right → 朝右 (5步)

2. 移动操作
   输入任意非方向字符 → 朝当前方向走 1 格 →
   该字符留在**离开的那一格**，变为不可通行的墙。

3. Backspace
   退回历史栈上一格。
   - **消除**上一格的足迹（字符被清除）
   - **返还**步数（退一步还一步）
   - 代价：**用时增加**（影响评分面板的用时得分）
   - 方向指令也可回退（撤销整条方向词）
   - 起点不可回退
```

### 🔑 2.4 核心规则：自我足迹不可通行

```
玩家留下的字符是不可通行的**永久墙**。
不存在"输入匹配字符消除足迹"的机制。
走过的路就是封死了——地图只会越走越小。
```

### 2.5 预置字符（唯一可消除的元素）

```
地图上预置的字符（渲染为浅色/淡色，区别于深色的自我足迹）。
走到该格时：
  - 输入匹配字符 → 该格清空变空地 (.)，后续可穿过
  - 输入不匹配字符 → 正常通过，字符保留

这是游戏中唯一能让地图"变宽"的机制。
```

### 2.6 步数系统

| 操作 | 消耗 |
|------|------|
| 移动 1 格 | 1 步 |
| 拼 `up` | 2 步 |
| 拼 `left` | 4 步 |
| 拼 `down` | 4 步 |
| 拼 `right` | 5 步 |
| 触发交互点 `_` | 1 步 |
| backspace | -1 步（返还）|

### 2.7 胜利/失败条件

| 条件 | 结果 |
|------|------|
| 步数归零 | **失败** |
| 站在终点格 `★` 上 | **胜利** |
| 最后一步恰好踏入终点 | 胜利（允许零步到达） |

## 三、关卡流程与大厅（Hub）

### 3.1 游戏流程总览

```
首次进入 → 教学关卡1（纯移动）→ 回到大厅
                    ↓
            教学关卡2（转向）→ 回到大厅
                    ↓
            关卡3（引入 press）→ 回到大厅
                    ↓
     大厅中用 press 开门 → 关卡4（引入 p_int）
                    ↓
             ...以此类推
```

### 3.2 大厅设计理念

大厅本身就是一个**可玩的文字狱关卡**。它不是菜单列表——它是一张地图，玩家用同样的移动/打字机制在其中行走，走到对应关卡的入口即进入该关。

```
大厅示例（草图）：

     ┌─────────────────────────┐
     │  ★1  .  .  .  ★2       │
     │           #             │
     │           #             │
     │           #             │
     │  @  .  .  p  .  .  ★3  │
     │            r            │
     │            e            │
     │            s            │
     │            _            │
     │            s            │
     │            ★4  .  .  . │
     └─────────────────────────┘

  @ = 玩家起始（每次回到大厅都在此）
  ★1 = 关卡1入口（已通关，显示为绿色★）
  ★2 = 关卡2入口（已通关）
  ★3 = 关卡3入口（已通关）
  ★4 = 关卡4入口 → 被 press 锁住，需触发开门
  p r e s s 中的 _ 输入 s 即可开门通往 ★4
```

### 3.3 渐进式机制教学

关卡顺序的设计遵循**机制逐渐叠加**的原则，每个新机制先单独教学，再在后续关卡中组合：

| 关卡 | 引入机制 | 教学目的 |
|------|---------|---------|
| 关卡1 | 基础移动 | 按字符朝右走，到达终点 |
| 关卡2 | 转向 | 拼 `up` `right` 改变朝向绕过墙 |
| 关卡3 | press 交互 | 走到 `_` 输入 `s` 触发开门 |
| 关卡4 | 传送门 | 从 `O` 进入从 `●` 出现 |
| 关卡5 | p_int 记录点 | 触发记录点后可按 R 回溯 |
| 关卡6 | 组合 | press + 传送门 + 记录点综合利用 |
| 关卡7 | lo_k 锁 | 先触发 lo_k 再通过锁门 |
| 关卡8+ | 综合谜题 | 所有机制混用 |

### 3.4 大厅中的"锁"机制

大厅中的道路会随着玩家通关进度逐渐开放：

```
刚进入游戏 → 只有 ★1 可到达（直接走入教学关）
通关关卡1 → 回到大厅，通向 ★2 的路开放
通关关卡2 → 回到大厅，通向 ★3 的路开放
通关关卡3 → 回到大厅，但通向 ★4 的路被 press 锁挡住
           → 玩家需要在已经走过的大厅中找路去触发 press
           → 开门后进入关卡4
```

这意味着大厅本身也在随手关卡的推进而"解锁新区域"，玩家每次回到大厅都有新的可探索空间。

### 3.5 关卡入口的渲染状态

| 状态 | 显示 | 行为 |
|------|------|------|
| 未解锁 | `★`（灰色） | 不可通行，路径被锁或墙挡住 |
| 可进入 | `★`（白色闪烁） | 走到上面即进入该关卡 |
| 已通关 | `★`（金色，带✓标记） | 可再次进入（重玩），也可绕过 |

### 3.6 关卡结束后的流程

```
通关 → 评分面板显示 → [下一关] 回到大厅，对应入口变为金色
                    → [重试]  重新开始当前关卡
                    → [退出]  回到大厅

失败 → 评分面板显示（失败不评分，仅展示步数和用时）
     → [重试]  重新开始
     → [退出]  回到大厅
```

## 四、评分系统

### 4.1 步数效率得分

| 计算方式 | 说明 |
|---------|------|
| `optimal` | 算法自动算出理论最优步数（见下方说明） |
| 得分 = `max(0, 100 - (实际步数 - optimal) × 2)` | 每多余一步扣 2 分，扣完为止 |

额外加分：实际步数 ≤ optimal → **额外 +20 分**（完美通关奖励）

**最优步数自动计算：**
```
状态空间: (x, y, direction)   → 4 朝向 × 可达格子数
移动成本: 朝当前方向走 1 格   = 1 步
转向成本: up=2 / down=4 / left=4 / right=5 步
目标:     (goal_x, goal_y, 任意朝向)

算法: Dijkstra（边权为正且≠1，不适合 BFS）
      状态数通常 < 2000，每帧重算也没压力
```

注意：算法只计算**静态地图**上的理论最优。玩家自己留下的足迹是动态变化的，所以在真实游戏中实际步数必然会大于等于这个理论值。

### 4.2 用时得分

| 计算方式 | 说明 |
|---------|------|
| 游戏内计时，从第一步输入开始到站在终点为止 |
| 得分 = `max(0, 100 - elapsed_seconds × 2)` | 每多一秒扣 2 分 |

### 4.3 🔑 文字质量得分（核心特色）

玩家通关时，将**历史输入序列**作为字符串进行可读性评分：

```rust
评分维度：
  1. 单词匹配          —— 输入串中包含了多少英文单词（可达 50 分）
  2. 句子结构          —— 是否包含 "I/we/you" 等主语 + 谓语结构（可达 30 分）
  3. 字符利用率        —— 有效单词字符占比（防止乱凑）（可达 20 分）

评分方法（轻量实现方案）：
  - 内置一个常用英文单词列表（~500-1000 个高频词）
  - 将输入序列按空格、标点或固定窗口分词后做字典匹配
  - 检测简单的语法模式（如 "i_verb_the"、"we_go_to"）
```

#### 评分示例

| 输入序列 | 单词分 | 句子分 | 利用率分 | 总分 | 评价 |
|---------|--------|--------|---------|------|------|
| `aaaaaaaa` | 0 | 0 | 0 | 0 | 无意义乱按 |
| `uuuuuuuu` | 0 | 0 | 0 | 0 | 原地转向怪 |
| `weup` | 10 (`we`) | 5 (主语) | 50% | 15+ | 有点意思 |
| `gouprightwego` | 20 (`go`, `up`, `right`, `we`, `go`) | 10 | 62% | 30+ | 可读性不错 |
| `we got his car` | 40 (4个单词) | 25 | 100% | 65+ | 哦？竟然是一句话 |
| `i saw the red cat` | 50 (5个单词) | 30 | 100% | 80+ | 连起来是合理句子 |

> 注意：玩家在游戏中**并非刻意写作文**，而是为了通关自然按键。这个评分系统只是**奖励**那些在无意中或有意中打出漂亮句子的玩家。负分不存在，最低 0 分。

### 4.4 总分

```
总分 = 步数得分 × 权重 + 用时得分 × 权重 + 文字质量得分 × 权重

示例权重：步数 30% + 用时 20% + 文字质量 50%
（文字质量权重最高，因为它是本游戏的特色所在）
```

## 五、地图元素

### 5.1 生态分类

地图元素按"会不会收缩/扩展玩家活动空间"分三类：

| 类别 | 包含元素 | 对空间的影响 |
|------|---------|------------|
| **收缩类** | 自我足迹（所有玩家留下的字符） | 永久墙，地图越来越小 |
| **扩展类** | 预置字符（匹配消除变空地） | 唯一能让地图变大的手段 |
| **中性** | 道路、墙、起点、终点、步数补给、交互点、传送门、锁门 | 不改变边界 |

### 5.2 图元总表

| 符号 | 名称 | 可通行 | 说明 |
|------|------|--------|------|
| `.` | 道路 | ✅ | 基础可通行格 |
| `#` | 墙 | ❌ | 不可通行，撞墙播放低沉音效 |
| `@` | 起点 | ✅ | 玩家初始位置 |
| `★` | 终点 | ✅ | 到达即胜利 |
| `0`-`9` | 步数补给 | ✅ | 走上去获得对应步数，格子变空地（一次性） |
| `_` | 交互点 | ✅ | 走上去输入匹配字符触发效果 |
| `O` | 传送门入口 | ✅ | 踩上传送到对应出口 |
| `●` | 传送门出口 | ✅ | 接收传送 |
| `✕` | 记录点标记 | ✅ | 标记记录点位置（仅在 p_int 触发后显示） |
| `+` | 锁门 | ❌(默认) | 交互触发后变空地 |
| 浅色字母 | 预置字符 | ✅ | 匹配输入可消除变空地，唯一扩展类元素 |
| 深色字母 | 自我足迹 | ❌ | 永久墙，地图收缩的根源 |

### 5.3 步数补给 `0`-`9`

```
走到数字格上 → 步数 += 数字值 → 该格变空地 (.)
特殊设计：
  0 是陷阱——消耗 1 步上去了，步数 +0，白走一格。
```

### 5.4 传送门 `O` / `●`

```
一进一出配对。从某方向进入 O →
从 ● 的对应反方向出现（保留进入时的朝向）。

例：
  O ← 从左边走进来（朝右）
  ● → 从右边出现（朝右，面朝出口方向）

传送不消耗额外步数（走上去那 1 步已扣）。
```

### 5.5 交互词系统（核心特色机制）

#### 概念

某些单词**纵向嵌入地图**，每个字母占一格且为**墙**，用特殊颜色（青色）渲染提示。单词中留出一个 `_` 作为路径上的交互点：

```
#######p#######
#######r#######
#######e#######
#######s#######
_______s_______    ← 路径在此通过，_ 是交互格
#######s#######
```

#### 交互词列表

| 地图嵌入 | 单词来源 | 触发字符 | 效果 |
|---------|---------|---------|------|
| `p r e s s` | press | `s` | 激活机关/开启对应门 |
| `p _ i n t` | point | `o` | 激活记录点（此后可按 R 回溯到此处） |
| `l o _ k` | lock | `c` | 解锁对应的锁门 `+` |

#### 交互流程

```
1. 玩家从路径走到 _ 格
2. 站在 _ 上，输入字符：
   ✅ 正确（如 s）→ _ 变绿色，效果触发
   ❌ 错误（如 x）→ _ 变红色，效果不触发（可自由离开）
3. 无论成功与否，无需"打完整个单词"，自由离开即可
4. 不需要重置等待，因为不触发也可以推进游戏
```

#### 引导色

```
单词字母（p, r, e, s...）：
  - 青色墙 → 提示"这里有可交互的东西"
  - 区别于普通 # 墙的灰色
_ 交互格：
  - 默认：青色闪烁/高亮
  - 触发成功：绿色
  - 触发失败：红色（下一帧或移动后恢复默认）
```

### 5.6 锁门 `+`

```
受 lo_k 交互控制的特殊墙。
lo_k 触发后 → 对应的 + 变空地 (.)。
```

## 八、技术与工程架构

### 6.1 技术选型

| 层 | 选择 | 理由 |
|----|------|------|
| 游戏引擎 | bracket-lib (0.8.x) | 轻量、Console/Terminal 原生支持、有 ECS |
| 渲染后端 | OpenGL (默认) | bracket-lib 内置 |
| 输入处理 | bracket-lib 的 `input` / `key` | 直接轮询 |
| 音效 | bracket-lib 内置的 audio / `bracket_audio` | 走一步响一声的粒度刚好 |
| 关卡序列化 | 纯文本 + 行解析 | 最直观，可手写地图 |

### 6.3 Tile 类型枚举

```rust
#[derive(Clone, Copy, PartialEq)]
enum Tile {
    Floor,                    // .
    Wall,                     // #
    Start,                    // @
    Goal,                     // ★
    StepBonus(u8),            // 0-9
    Interactive(&'static str, char, InteractiveState), // _ 交互点（所属词、触发字符、状态）
    PortalEntry,              // O
    PortalExit(u32),          // ●（配对 ID）
    Checkpoint,               // ✕
    LockedDoor,               // +
    Preplaced(char),          // 预置字符（浅色渲染）
    Footprint(char),          // 自我足迹（深色渲染，永久墙）
}
```

### 6.4 游戏主循环

```rust
fn tick(&mut self, ctx: &mut BTerm) {
    // 1. 处理输入（仅在非动画状态时）
    if let Some(key) = ctx.key {
        match key {
            VirtualKeyCode::Back => try_undo(),
            c if self.standing_on_interactive() => try_interact(c),
            c if is_direction_char(c) => feed_direction_buffer(c),
            c => try_move(c),
        }
    }

    // 2. 更新状态
    flush_direction_buffer();   // 检出方向词是否完整
    check_steps();              // 步数归零 → 失败
    check_goal();               // 终点 → 胜利

    // 3. 渲染
    draw_map(ctx);
    draw_player(ctx);
    draw_ui(ctx);
}
```

### 6.6 交互触发优先级

```
站在 _ 上时：
  1. 输入的字符优先匹配该交互点的触发字符
  2. 匹配 → 触发效果，不移动
  3. 不匹配 → 变红提示，不移动
  4. 不参与方向词累积，不走动

不站在 _ 上时：
  1. 是方向相关字符 → 馈入方向缓冲区
  2. 方向词完整匹配 → 转向
  3. 普通字符 → 朝当前方向移动 1 格
```

## 十、设计哲学

### 核心矛盾：扩张 vs 收缩

```
     ┌─────────────────────┐
     │     预置字符         │
     │  （唯一可扩张路径）  │
     │        ↓            │
     │ 地图空间在变化       │
     │        ↓            │
     │   自我足迹           │
     │  （永久收缩路径）    │
     └─────────────────────┘
```

- 每一步都在**消耗地图空间**（自我足迹永久封路）
- 只有预置字符提供**有限的路径扩展**
- 玩家紧张感来源于"我能走的空间正在不可逆地缩小"
- 策略感来源于"什么时候该消耗步数去消除预置字符打开通道"

### Backspace 的特殊意义

```
backspace 退回但不消除足迹——意味着：
  "你可以后悔，但不能抹去历史"
  退回让你可以换一条路走，但之前铺满字符的岔路已经封死了
```
